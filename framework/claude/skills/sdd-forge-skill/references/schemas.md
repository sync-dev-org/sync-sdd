# JSON Schemas

Schema definitions for all JSON files used in the eval/benchmark pipeline. These schemas are the contract between scripts, SubAgents, and the HTML viewer.

## evals.json

Eval definitions. Created by the user (with Lead's help) or generated by scripts.

```json
{
  "evals": [
    {
      "id": "string (unique identifier, e.g. eval-001)",
      "prompt": "string (the user message that triggers the skill)",
      "expected_output": "string (human-readable description of expected behavior)",
      "files": ["string (paths to input files the eval needs)"],
      "expectations": [
        {
          "description": "string (what should happen)",
          "criterion": "string (how to verify it happened)"
        }
      ]
    }
  ]
}
```

**Notes:**
- `id` must be unique within the file
- `files` are relative to the project root; they are made available to the execution environment
- `expectations` are graded by the grader SubAgent; each becomes a PASS/FAIL row in the benchmark

## history.json

Version progression tracking, used in improve mode to track how a skill evolves across iterations.

```json
{
  "versions": [
    {
      "version": "integer (1-indexed)",
      "timestamp": "string (ISO-8601)",
      "mode": "string (create | improve | reforge)",
      "description_length": "integer (character count)",
      "body_lines": "integer (SKILL.md line count)",
      "changes": "string (summary of what changed)",
      "benchmark_pass_rate": "number (0.0-1.0, optional)",
      "feedback": "string (user feedback that prompted this version, optional)"
    }
  ]
}
```

## grading.json

Output from the grader SubAgent. One file per eval execution.

**Viewer-critical field names**: The expectations array MUST use `text`, `passed` (boolean), and `evidence`. The HTML viewer and `aggregate_benchmark.py` depend on these exact names. Using `description`/`result`/`name`/`met` will break the pipeline.

```json
{
  "expectations": [
    {
      "text": "string (the assertion text)",
      "passed": "boolean (true | false)",
      "evidence": "string (specific citation from transcript/output)"
    }
  ],
  "summary": {
    "total": "integer",
    "passed": "integer",
    "failed": "integer",
    "pass_rate": "number (0.0-1.0)"
  },
  "execution_metrics": {
    "tool_calls": {
      "Read": "integer",
      "Write": "integer",
      "Bash": "integer"
    },
    "total_tool_calls": "integer",
    "files_created": ["string"],
    "errors_encountered": "integer"
  },
  "timing": {
    "total_tokens": "integer",
    "duration_ms": "integer",
    "total_duration_seconds": "number"
  },
  "claims": [
    {
      "claim": "string",
      "type": "string (factual | process | quality)",
      "verified": "boolean | string (true | false | unverifiable)",
      "evidence": "string"
    }
  ],
  "user_notes_summary": "string (optional)",
  "eval_feedback": {
    "too_easy": ["string"],
    "no_coverage": ["string"],
    "unverifiable": ["string"],
    "suggestions": ["string"]
  }
}
```

## metrics.json

Execution metrics captured by the eval executor. Located at `<run-dir>/outputs/metrics.json`.

```json
{
  "tool_calls": {
    "Read": "integer",
    "Write": "integer",
    "Bash": "integer",
    "Edit": "integer",
    "Glob": "integer",
    "Grep": "integer"
  },
  "total_tool_calls": "integer",
  "total_steps": "integer",
  "files_created": ["string (filenames)"],
  "errors_encountered": "integer",
  "output_chars": "integer (total character count of output files)",
  "transcript_chars": "integer"
}
```

## timing.json

Wall clock timing data from task notifications. When a subagent task completes, the notification includes `total_tokens` and `duration_ms`. Save these immediately — they are not persisted elsewhere.

```json
{
  "total_tokens": "integer",
  "duration_ms": "integer",
  "total_duration_seconds": "number"
}
```

## benchmark.json

Aggregated benchmark results. The HTML viewer and `aggregate_benchmark.py` depend on exact field names — do not rename fields or change configuration values.

```json
{
  "metadata": {
    "skill_name": "string",
    "skill_path": "string",
    "timestamp": "string (ISO-8601)",
    "evals_run": ["integer or string (eval IDs)"],
    "runs_per_configuration": "integer (e.g. 3)"
  },
  "runs": [
    {
      "eval_id": "integer or string",
      "eval_name": "string (human-readable, used as section header in viewer)",
      "configuration": "string (with_skill | without_skill)",
      "run_number": "integer (1-indexed)",
      "result": {
        "pass_rate": "number (0.0-1.0)",
        "passed": "integer",
        "failed": "integer",
        "total": "integer",
        "time_seconds": "number",
        "tokens": "integer",
        "tool_calls": "integer",
        "errors": "integer"
      },
      "expectations": [
        {"text": "string", "passed": "boolean", "evidence": "string"}
      ],
      "notes": ["string (per-run observations)"]
    }
  ],
  "run_summary": {
    "with_skill": {
      "pass_rate": {"mean": "number", "stddev": "number", "min": "number", "max": "number"},
      "time_seconds": {"mean": "number", "stddev": "number", "min": "number", "max": "number"},
      "tokens": {"mean": "number", "stddev": "number", "min": "number", "max": "number"}
    },
    "without_skill": {
      "pass_rate": {"mean": "number", "stddev": "number", "min": "number", "max": "number"},
      "time_seconds": {"mean": "number", "stddev": "number", "min": "number", "max": "number"},
      "tokens": {"mean": "number", "stddev": "number", "min": "number", "max": "number"}
    },
    "delta": {
      "pass_rate": "string (e.g. '+0.50')",
      "time_seconds": "string (e.g. '+13.0')",
      "tokens": "string (e.g. '+1700')"
    }
  },
  "notes": ["string (auto-generated analyst observations)"]
}
```

**Viewer-critical field names (do not change):**
- `configuration`: must be `"with_skill"` or `"without_skill"` (viewer uses these literal strings for grouping and color coding; `"treatment"/"baseline"` will NOT work)
- `run_number`: 1-indexed integer (viewer displays this directly)
- `eval_name`: used as section header in per-eval breakdown
- `result.pass_rate`: nested under `result` (not top-level on the run) — used for chart rendering
- `result.passed`, `result.total`: used for "X/Y" display
- `expectations[].text`, `expectations[].passed`, `expectations[].evidence`: per-assertion viewer
- `run_summary.{with_skill,without_skill}.pass_rate.{mean,stddev}`: used for summary table
- `run_summary.delta.pass_rate`: used for delta display
- `metadata.runs_per_configuration`: used in report header

## analysis.json

Output from the post-hoc analyzer SubAgent (comparison analysis role).

```json
{
  "comparison_summary": {
    "winner": "string (A | B)",
    "eval_prompt": "string",
    "key_finding": "string"
  },
  "instruction_following": {
    "a_score": "integer (1-10)",
    "a_notes": "string",
    "b_score": "integer (1-10)",
    "b_notes": "string"
  },
  "strengths": [
    {
      "skill": "string (A | B)",
      "aspect": "string",
      "evidence": "string",
      "impact": "string (high | medium | low)"
    }
  ],
  "weaknesses": [
    {
      "skill": "string (A | B)",
      "aspect": "string",
      "evidence": "string",
      "impact": "string (high | medium | low)"
    }
  ],
  "suggestions": [
    {
      "priority": "string (high | medium | low)",
      "type": "string (instructions | tools | examples | error_handling | structure | references)",
      "target": "string (A | B)",
      "suggestion": "string",
      "reasoning": "string"
    }
  ],
  "transcript_insights": {
    "a_notable": ["string"],
    "b_notable": ["string"],
    "patterns": "string"
  }
}
```

  "assertion_patterns": [
